# Source Catalog Schema

`SourceCatalog` is the public support matrix for providers and official sources
that APW currently watches.

The canonical artifact is:

```text
data/feeds/source-catalog.json
```

It is generated by `apw index`, bundled in Python package data, included in
release dry-run artifacts, and listed in `data/releases/*/checksums.txt`.

## Purpose

Consumers use the source catalog to answer:

- which providers APW currently covers;
- which official source pages, feeds, docs, and status surfaces are registered;
- what type of changes each source is expected to reveal;
- whether a source is enabled for deterministic fetching;
- when source state was last fetched and validated;
- how many parser fixtures protect each source parser;
- how many reviewed events cite each source;
- whether review candidates are waiting for source-owner action.

`SourceCatalog` is a discovery and trust surface. It does not publish provider
events, and it does not grant source-refresh, event-promotion, tag, package, or
release authority.

## Contract

- `schema_version`: `apw.source_catalog.v0`.
- `summary`: provider, source, validation, cadence, source-type, event-kind,
  reviewed-event, latest-event, backlog, and source-state counts.
- `providers`: provider display names, homepage URLs, aliases, source counts,
  validated-source counts, source types, cadence hints, impact hints,
  reviewed-event counts, event-kind counts, source keys, and provider surfaces.
- `sources`: source key, provider refs, source type, authority, URL, cadence,
  enabled flag, automation status, parser, validation status, validation
  timestamp, optional adoption-history fields, parser-fixture count,
  reviewed-event count, latest reviewed-event date, backlog count, policies,
  maintainers, and public source-state metadata.
- `consumer_guidance`: short guidance for the live reviewed feed, immutable
  data tags, installed package snapshots, and the review boundary.
- `catalog_policy`: explicit statement that the artifact contains no raw
  provider content.

`introduced_at` and `introduced_ref` are optional source-adoption history
fields. They are available for new source descriptors, but historical rows may
remain `null` until maintainers backfill them from PR history.

`last_validated_at` means APW has committed source-state metadata from a source
fetch. It is not a promise that every provider page row has become a reviewed
event. Candidate generation remains review-only, and source owners promote only
specific official facts into `data/events/*.json`.

## CLI

Print JSON:

```bash
apw source catalog
```

Print a compact human summary:

```bash
apw source catalog --summary
```

Fetch the current reviewed catalog from GitHub without a checkout:

```bash
apw remote feed source-catalog --ref main --output apw-source-catalog.json
```

Validate that catalog metadata is present, schema-valid, and current:

```bash
apw validate
apw index --check
```
