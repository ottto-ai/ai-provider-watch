# v0.2 Release Checklist

Use this checklist before cutting an APW `v0.2.x` package, publishing a
`data-YYYY.MM.DD` data tag, or declaring the v0.2 operations loop ready. It
collects the existing runbooks into one release-manager packet; it does not
grant automation new publishing authority.

## Release Scope

Record these facts in the release PR or evidence packet:

- release type: package, data tag, or both;
- source commit and branch;
- target package version, when applicable;
- target data tag, when applicable;
- source-owner approval for any reviewed event, source descriptor, parser,
  registry, generated feed, or source-state change;
- release-manager approval from `@RonShub` during the v0.x single-maintainer
  period;
- no raw provider text, private Ottto surface, customer telemetry, credentials,
  screenshots, or authenticated-console data in the diff or artifacts.

## Local Release Gates

Run from a clean checkout of the intended release commit:

```bash
uv sync --all-extras
uv lock --check
uv run ruff check .
uv run pytest
uv run apw source test
uv run apw source coverage --summary
uv run apw operations report --summary
uv run apw operations launch-gate --summary
uv run apw validate
uv run apw index --check
uv run apw freshness --summary
actionlint .github/workflows/*.yml
uv run apw release dry-run --output .apw/release-dry-run --require-clean
uv run apw release packet --dry-run-report .apw/release-dry-run/data-YYYY.MM.DD/dry-run-report.json ...
git diff --check
```

For package releases, also run:

```bash
uv build --out-dir .apw/dist
python -m venv /tmp/apw-package-smoke
. /tmp/apw-package-smoke/bin/activate
python -m pip install --upgrade pip
python -m pip install .apw/dist/ai_provider_watch-*.whl
apw --root "$PWD" validate
apw --root "$PWD" index --check
apw --root "$PWD" freshness --summary
```

For installed package-data mode, run from outside the checkout:

```bash
mkdir -p /tmp/apw-installed-data-smoke
cd /tmp/apw-installed-data-smoke
apw validate
apw index --check
apw freshness --summary
apw latest --limit 3
apw diff --since 30d
apw explain 2026-06-01-google-vertex-gemini-2-0-flash-retirement
```

## Candidate And Event Review

Before any data tag includes new provider facts:

- review candidate JSON as untrusted data;
- confirm official evidence URLs are public, unauthenticated, and inside source
  descriptor `allowed_domains`;
- promote only reviewed facts into `data/events/`;
- regenerate feeds, indexes, freshness, and release manifests with `apw index`;
- record rejected, duplicate, superseded, or split candidates in the PR body or
  local ignored review notes;
- confirm candidate generation, source refresh, LLM review, PR-comment, MCP,
  issue, social, and downstream jobs had no release token, OIDC publishing
  authority, tag authority, PyPI token, or release-upload authority;
- record release-token separation explicitly in the release evidence packet.

## Agent And MCP Gates

Keep every agent-facing lane read-only or review-only:

- `uv run pytest tests/test_prompt_injection_redteam.py`;
- `uv run pytest tests/test_mcp_readonly.py`;
- `uv run pytest tests/test_llm_review.py`;
- render an LLM review request only as a bounded packet with omitted
  `claim_text`, claim hashes, and forbidden actions;
- if Codex or Vertex Gemini Flash is used for review, store the model output as
  advisory review input only;
- do not let MCP resources or tool outputs publish events, mutate sources,
  merge PRs, create tags, request OIDC, or read release credentials.

## Downstream Adoption Gates

For v0.2, downstream examples must keep working without an Ottto account:

- `uv run pytest tests/test_downstream_smoke_fixtures.py`;
- verify LiteLLM, models.dev, Langfuse, Helicone, OpenLIT, Codex, Claude Code,
  Cursor, Copilot, gateway, webhook, and Slack-style examples are schema-valid;
- confirm examples use event IDs, checksums, freshness metadata, idempotency
  keys, and read-only tokens where tokens are needed;
- confirm examples do not include private Ottto endpoints, source-line leakage,
  third-party API keys, write scopes, release tokens, or provider credentials.

## GitHub And Security Gates

Record exact command output or URLs:

```bash
export REPO=ottto-ai/ai-provider-watch
export SHA="$(git rev-parse HEAD)"

gh api "repos/$REPO/branches/main/protection" \
  --jq '{required_status_checks, required_pull_request_reviews, enforce_admins}'

gh api "repos/$REPO/actions/permissions/workflow" \
  --jq '{default_workflow_permissions, can_approve_pull_request_reviews}'

gh api "repos/$REPO" \
  --jq '{security_and_analysis, delete_branch_on_merge, allow_squash_merge}'

gh run list --repo "$REPO" --branch main --commit "$SHA" \
  --workflow CI --json status,conclusion,url,headSha

gh run list --repo "$REPO" --branch main --commit "$SHA" \
  --workflow CodeQL --json status,conclusion,url,headSha

gh workflow run dependency-review.yml --repo "$REPO" --ref main \
  -f base_ref="<previous-release-tag-or-sha>" \
  -f head_ref="$SHA"

gh run list --repo "$REPO" --workflow "Dependency Review" \
  --json status,conclusion,url,headSha
```

Release is blocked if required checks are not green, branch protection or a
ruleset is absent, Dependency Review cannot run, CodeQL analysis is missing for
the release commit, repository security settings are not recorded, Scorecard is
missing or failing, or workflow permissions drift from read-only defaults for
source-processing jobs.

During v0.x, `main` may use zero required approving reviews to avoid a
single-maintainer self-review deadlock. That exception is acceptable only while
strict required checks remain enabled and `@RonShub` is the sole release
manager, source owner, schema maintainer, and security contact. Revisit this
before v1.0 or before adding another release manager.

## Data Tag Packet

For a real `data-YYYY.MM.DD` tag:

1. Confirm the release commit is on `main`.
2. Verify local release dry-run artifacts, `data/feeds/freshness.json`, and
   `data/feeds/operations.json`.
3. Download the GitHub dry-run artifact bundle.
4. Verify checksums against the generated manifest.
5. Verify the artifact attestation:

   ```bash
   gh attestation verify .apw/apw-release-dry-run.tgz --repo "$REPO"
   ```

6. Render and review the publication packet:

   ```bash
   uv run apw release packet \
     --dry-run-report .apw/release-dry-run/data-YYYY.MM.DD/dry-run-report.json \
     --reviewed-event "<event-id>" \
     --release-manager @RonShub \
     --source-owner @RonShub \
     --source-owner-approval-ref "<approval-url>" \
     --release-manager-approval-ref "<approval-url>" \
     --branch-protection-ref "<branch-protection-ref>" \
     --ci-ref "<ci-run-url>" \
     --codeql-workflow-ref "<codeql-run-url>" \
     --code-scanning-ref "<analysis-ref>" \
     --dependency-review-ref "<dependency-review-run-url>" \
     --scorecard-ref "<scorecard-run-url>" \
     --attestation-ref "<attestation-verify-ref>" \
     --checksum-review-ref "<checksum-review-ref>" \
     --output .apw/release-dry-run/data-YYYY.MM.DD/publication-packet.json
   ```

   If there are no source-owner-reviewed event changes for the date, use
   `--allow-no-reviewed-events` and a specific `--skip-reason` instead of
   creating an empty data tag.

7. Create and verify the manual signed tag:

   ```bash
   git tag -s data-YYYY.MM.DD
   git tag -v data-YYYY.MM.DD
   git push origin data-YYYY.MM.DD
   ```

8. Publish the matching GitHub data release with the release evidence packet.

Do not store signing keys in GitHub Actions, repository secrets, environment
secrets, or workflows that process provider/source/candidate text.

## Python Package Packet

For a real package release:

1. Confirm the PyPI Trusted Publisher is configured for project
   `ai-provider-watch`, owner `ottto-ai`, repository `ai-provider-watch`,
   workflow `publish-python.yml`, and environment `pypi`.
2. Confirm the GitHub `pypi` environment requires `@RonShub` review during
   v0.x.
3. Build and publish only through PyPI Trusted Publishing; do not add PyPI API
   tokens or package-release secrets.
4. Create and push a signed package tag such as `v0.2.0`.
5. Approve the protected `pypi` environment deployment.
6. Verify PyPI project metadata, file hashes, and attestations.
7. Attach the exact PyPI wheel and sdist artifacts to the matching GitHub
   release; do not rebuild artifacts for GitHub.
8. Run the install and installed package-data smokes above.

## Rollback And Incident Plan

- never delete and recreate a Git tag, PyPI version, or data tag;
- prefer a patch release for ordinary package bugs;
- yank a PyPI release only for install/startup failure, private data, raw
  provider text, credentials, materially wrong data that cannot wait for a
  patch, or a security issue;
- for bad data, publish a corrected event/feed release or a superseding
  retraction event, then record the impact and mitigation;
- keep GitHub releases visible with clear yanked or superseded notes;
- open a public issue with impact, mitigation, replacement version or data tag,
  and source-owner/release-manager decision.

## Closeout

Before marking v0.2 ready:

- update `README.md`, `ROADMAP.md`, and operation docs if commands or promises
  changed;
- update GitHub milestone issues and PRs;
- update the private execution board;
- record evidence paths, commands, workflow URLs, tag IDs, package hashes,
  unresolved decisions, and next automation recommendation in the final report.
